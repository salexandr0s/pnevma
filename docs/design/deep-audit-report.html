Pnevma — Deep Security Audit

Date: 2026-03-05 · Target: pnevma v0.1.0 · Auditor: Claude Opus 4.6 · Classification: Internal

2 High 8 Medium 4 Low 1 Info 1 Advisory (Dep)

Executive Summary

Pnevma is a Rust + Swift/AppKit desktop application that orchestrates CLI coding agents (Claude Code, Codex), manages terminal sessions, git worktrees, SSH keys, and exposes a remote access server over Tailscale. The audit covers 10 Rust crates (~80 source files), 35+ Swift files, 13 DB migrations, CI/CD pipeline, release scripts, and the FFI bridge.

No critical vulnerabilities were found. Two high-severity issues relate to (1) missing HTTP body size limits on the remote server and (2) --dangerously-skip-permissions being passed to agents with worktree-only isolation. Eight medium findings span auth gaps, input injection surfaces, secret handling, and FFI safety. The codebase shows strong defensive posture overall: parameterized SQL, no shell string concatenation, panic-safe FFI, Argon2id password hashing, and structured rate limiting.

High Severity

F1 HIGH No HTTP request body size limit on remote server pnevma-remote/src/server.rs

OWASP: A05 Security Misconfiguration

Evidence: The axum router in server.rs does not add a RequestBodyLimit layer. While axum's Json extractor has a default 2MB limit, this is undocumented/version-dependent and does not cover raw body handlers or WebSocket upgrade paths.

Impact: An authenticated attacker can send arbitrarily large JSON bodies to POST /api/rpc, POST /api/tasks, or POST /api/sessions/:id/input, causing unbounded memory allocation and potential OOM crash (DoS).

Fix: Add explicit tower_http::limit::RequestBodyLimitLayer::new(2 * 1024 * 1024) to the router. Consider lower limits for specific endpoints.

// server.rs — add to router builder
.layer(tower_http::limit::RequestBodyLimitLayer::new(2 * 1024 * 1024))

ASVS: V14.1.1 — Verify that the application restricts the size of requests.

Owner: pnevma-remote maintainer · Retest: Send 10MB POST to /api/rpc; verify 413 response.

F2 HIGH --dangerously-skip-permissions with worktree-only isolation pnevma-agents/src/adapters/claude.rs:89, commands/tasks.rs:1520

OWASP: A05 Security Misconfiguration · A01 Broken Access Control

Evidence: When auto_approve: true (currently hardcoded for all dispatched tasks), Claude Code is invoked with --dangerously-skip-permissions. The code comment acknowledges: "worktree isolation is the safety boundary."

// claude.rs:89
args.push("--dangerously-skip-permissions".into());

// tasks.rs:1520
auto_approve: true, // worktree isolation is the safety boundary

Impact: Git worktrees share the same filesystem and user permissions. A compromised agent (via prompt injection or malicious task) can: read arbitrary files outside the worktree (~/.ssh/, ~/.aws/, Keychain DB), exfiltrate environment variables containing API keys, write to files outside the worktree, or spawn network connections. The worktree only constrains the CWD, not the process capabilities.

Fix: (1) Make auto_approve configurable in pnevma.toml with a default of false. (2) Document the risk tradeoff clearly. (3) Consider macOS sandbox profiles or container isolation for agent processes. (4) At minimum, strip sensitive environment variables before agent spawn.

ASVS: V1.4.2 — Verify access controls enforced at a trusted level, V4.1.1 — Principle of least privilege.

Owner: pnevma-agents maintainer · Retest: Dispatch a task with a prompt that reads ~/.ssh/id_ed25519; verify the agent is blocked.

Medium Severity

F3 MEDIUM session.send_input in RPC allowlist enables stdin injection pnevma-remote/src/routes/rpc_allowlist.rs:14

OWASP: A01 Broken Access Control · A03 Injection

Evidence: session.send_input is whitelisted in the remote RPC allowlist. An authenticated remote user can write arbitrary bytes to any active tmux session's stdin — including running shell sessions and AI agents. No ownership check validates the session belongs to the user's current project.

Impact: An authenticated remote attacker can inject shell commands into any running terminal session, gaining arbitrary code execution on the host.

Fix: (1) Remove session.send_input from the remote RPC allowlist, or (2) add project-scoped session ownership validation before input delivery, and (3) implement input content filtering for control characters.

ASVS: V4.2.1 — Verify that sensitive data and APIs are protected against IDOR.

F4 MEDIUM No token revocation mechanism pnevma-remote/src/auth.rs

OWASP: A07 Identification & Authentication Failures

Evidence: TokenStore supports issue_token() and validate_token(), plus cleanup_expired(). No revoke_token() method exists. No logout/revocation API route.

Impact: A stolen token remains valid for its full TTL (24h default). No way to invalidate a compromised session short of restarting the server.

Fix: Add revoke_token(token_id) to TokenStore and expose DELETE /api/auth/token. Consider shorter TTLs (1h) with token refresh.

ASVS: V3.3.1 — Verify that logout and expiration invalidate tokens.

F5 MEDIUM Auth token accepted in URL query parameter pnevma-remote/src/middleware/auth_token.rs:40-51

OWASP: A02 Cryptographic Failures

Evidence: auth_token.rs falls back to ?token=<value> when no Authorization header is present. Tokens in query strings appear in browser history, Tailscale network logs, referrer headers, and intermediary access logs.

Impact: Token leakage through URL logging. Primarily affects WebSocket upgrade paths where headers may not be available in all clients.

Fix: Restrict query-string token to WebSocket upgrade requests only. Log a warning when this path is used. Prefer cookie-based auth for browser clients.

ASVS: V3.1.1 — Verify URL does not expose session tokens.

F6 MEDIUM CheckType::TestCommand executes arbitrary shell strings from DB pnevma-commands/src/commands/mod.rs:1954-1960

OWASP: A03 Injection

Evidence:

let out = TokioCommand::new("zsh")
    .arg("-lc")
    .arg(&command)
    .current_dir(&worktree_path)
    .output()
    .await

The command value comes from check.command in the task's acceptance criteria, stored in the DB. If task creation is accessible via the remote API (it is — task.create may be in the allowlist), this is a remote code execution path.

Impact: An attacker who can create tasks with custom acceptance criteria can execute arbitrary commands in the context of the worktree (and beyond, since the shell has full user permissions).

Fix: (1) Validate/whitelist test commands against a known set of patterns. (2) Ensure task.create is not in the remote RPC allowlist, or if it is, sanitize the acceptance criteria field. (3) Run test commands in a restricted sandbox.

ASVS: V5.3.8 — Verify OS command injection protections.

F7 MEDIUM Session output redaction passes empty secrets list pnevma-commands/src/commands/mod.rs:2638

OWASP: A02 Cryptographic Failures (data exposure)

Evidence: redact_text(&chunk, &[]) is called with an empty secrets slice. Structural patterns (e.g., token=<value>) are caught, but literal secret values (API keys injected as env vars) are not scrubbed from terminal output streamed to the remote API or persisted to scrollback files.

Impact: If an agent or shell command echoes an API key in a non-structural format (e.g., echo $ANTHROPIC_API_KEY), the value will be visible in scrollback, event logs, and remote WS streams.

Fix: Populate the secrets list from keychain entries and known environment variable values. Pass this list to all redact_text callsites.

ASVS: V8.3.1 — Verify sensitive data is masked in logs.

F8 MEDIUM Keychain secret value exposed in process arguments pnevma-commands/src/commands/mod.rs:1816-1829

OWASP: A02 Cryptographic Failures

Evidence: store_keychain_secret passes the secret value as a CLI argument to security add-generic-password -w <value>. Process arguments are visible in ps output.

TokioCommand::new("security")
    .args(["add-generic-password", "-U", "-s", service, "-a", account, "-w", value])

Impact: The secret is briefly visible (~10ms) in the process table to any process running as the same user or with elevated privileges.

Fix: Use the macOS Security framework API directly via FFI (e.g., security-framework Rust crate) instead of shelling out. Alternatively, pipe the password via stdin using -w without a value argument and providing it via stdin.

ASVS: V6.4.1 — Verify secrets not exposed in process artifacts.

F9 MEDIUM No WebSocket message size limit pnevma-remote/src/routes/ws.rs

OWASP: A05 Security Misconfiguration

Evidence: The WebSocket handler reads Message::Text(t) with no maximum size check before JSON parsing. A large message could cause memory pressure.

Impact: Authenticated attacker can send oversized WS messages to cause memory pressure or slow JSON parsing (DoS).

Fix: Configure axum WebSocket with max_message_size (e.g., 64KB). Reject messages exceeding the limit.

ASVS: V13.2.1 — Verify that WebSocket connections have appropriate limits.

F10 MEDIUM FFI callback pointer lifetime contract unenforced pnevma-bridge/src/lib.rs:268-283

OWASP: A06 Vulnerable Components (memory safety)

Evidence: In pnevma_call_async, cb_ctx is cast to usize and stored across an async boundary. The caller must guarantee the context pointer remains valid until the callback fires. No runtime enforcement exists.

let cb_ctx_usize = cb_ctx as usize;
handle.runtime.spawn(async move {
    ...
    cb(r, cb_ctx_usize as *mut ());
});

Impact: If the Swift caller frees or moves the context object before the async task completes, this results in a use-after-free. Also, pnevma_free_result is called automatically after the callback, creating a double-free risk if the Swift side also frees the result.

Fix: (1) Use Arc-wrapped context with explicit reference counting shared across the FFI boundary. (2) Add a sentinel/canary value to detect freed contexts at runtime. (3) Document the contract with #[safety] doc comments.

ASVS: V14.2.1 — Verify all components are free of known vulnerabilities.

Low Severity

F11 LOW Silent fallback to self-signed TLS certificate pnevma-remote/src/tls.rs:64-66

When Tailscale certificates are unavailable, the server silently falls back to a self-signed cert with only a tracing::warn!. Clients cannot pin this certificate, weakening MITM protection.

Fix: Emit the certificate fingerprint at startup. Provide a config option to reject self-signed mode. Display cert status in the UI.

F12 LOW Tailscale guard rejects IPv6 connections pnevma-remote/src/tailscale.rs:51-59

is_tailscale_ip returns false for all IPv6 addresses. Tailscale uses IPv6 (fd7a:115c:a1e0::/48) for newer deployments, causing legitimate clients to be locked out.

Fix: Add the Tailscale IPv6 CGNAT range (fd7a:115c:a1e0::/48) to the guard's allow list.

F13 LOW Token TTL comparison uses hours granularity pnevma-remote/src/auth.rs:65

(Utc::now() - entry.created_at).num_hours() truncates to whole hours. A token issued at 23:59 is considered 0 hours old until 00:59, granting up to ~2h extra validity beyond the configured TTL.

Fix: Use num_seconds() or direct DateTime comparison: Utc::now() > entry.created_at + Duration::hours(ttl).

F14 LOW ssh-keygen comment argument not validated pnevma-ssh/src/key_manager.rs:95-106

User-supplied comment is passed directly to ssh-keygen -C <comment>. While Command::new prevents shell injection, null bytes or very long strings could cause unexpected behavior in ssh-keygen.

Fix: Validate comment: max 256 chars, printable ASCII only, strip null bytes.

Informational

F15 INFO Protected action confirmation not enforced in audited Rust code pnevma-core/src/protected_actions.rs

protected_actions.rs defines ActionKind, RiskLevel, and confirmation phrases, but contains no enforcement logic. Actual enforcement must happen in the Swift UI layer (ProtectedActionSheet.swift). Verify that the UI always consults this module before executing destructive actions.

Fix: Add a verify_confirmation(action: ActionKind, phrase: &str) -> bool function to the Rust side and call it from the bridge before executing protected operations.

Dependency Advisory

D1 ADVISORY RUSTSEC-2023-0071: Marvin Attack in rsa crate (via sqlx-mysql) Cargo.lock: rsa 0.9.10 ← sqlx-mysql 0.8.6

Severity: CVSS 5.9 (Medium) — potential key recovery through timing side channels.

Impact on Pnevma: Low/None. Pnevma uses SQLite only (not MySQL). The rsa crate is pulled in transitively by sqlx-mysql, which is compiled but never used at runtime. No fix is currently available upstream.

Mitigation: Disable the mysql feature in sqlx if possible: sqlx = { version = "0.8", default-features = false, features = ["sqlite", "runtime-tokio-rustls", ...] }. This is already partially done (default-features = false), but sqlx-mysql may be pulled in by the migrate or macros features.
